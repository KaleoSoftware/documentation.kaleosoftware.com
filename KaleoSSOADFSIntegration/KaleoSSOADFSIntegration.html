Overview

Kaleo can integrate seamlessly into your company’s existing Microsoft Active Directory infrastructure.

K

aleo is delivered as a hosted service via the cloud, and has no on-premise components. However, your Kaleo tenant instance can be configured to authenticate users against your existing ActiveDirectory (AD) infrastructure, and provide your users with a frictionless experience when accessing Kaleo. When operated in this mode, a User Account cannot be created in Kaleo, and all Users must come from AD.

This is accomplished by leveraging Microsoft’s Active Directory Federation Service (ADFS 2.0).  Once this is installed into your AD environment, clients can authenticate using the industry standard SAML protocol.  The way this typically works, is that the user’s browser attempts to access a restricted resource in the Kaleo webapp, which then redirects the browser to the standard ADFS endpoint at https://example.com/adfs/ls/  Based on the referrer and some information sent on the query string, ADFS will know that Kaleo is making the request for authentication, and will attempt to authenticate the user. If the user has already logged into the domain, and is using a supported browser, ADFS will automatically authenticate them, and redirect the browser back to Kaleo, along with an encrypted payload with the user’s email address and optionally other information about the user, which Kaleo will them rely on to automatically create a new Kaleo account for the user or log them into an existing Kaleo account.

Configuring ADFS Web Service

First, install the ADFS 2.0 software into your ActiveDirectory infrastructure. You will need to go through the ADFS 2.0 Federation Server Configuration Wizard and set up the system as appropriate for your environment. This will involve ensuring that the IIS server that AD FS is running on has a valid DNS name and SSL certificate. A self-signed certificate is fine for testing in a lab, but in a production environment it is important to have an actual SSL cert that will be trusted by all browsers. At this stage you should be able to visit the ADFS server in a browser and see a welcome to IIS page.

The next step is to lock down the IIS server to only allow authenticated users to access it.

Once you have done that, when you visit the ADFS web server using a browser that is not authenticated with the domain, like Firefox, you should receive a popup asking for your domain credentials.  When you visit the server using IE and are logged in to the domain, you should not be prompted for credentials, as ADFS will use your windows credentials to authenticate you.

Configuring ADFS for Kaleo

Now, we need to establish a trust relationship between Kaleo and ADFS.  In the ADFS 2.0 MMC snap-in select the certificates node and double click the token-signing certificate to view it. Copy the Thumbprint (also known as the idp_cert_fingerprint in SAML parlance) as this will be needed by your Kaleo integration specialist.

The next step is to add a Relying Party Trust

Once the Kaleo Software Relying Party entry has been created, right-click on it and go to Properties, and make sure the advanced tab has SHA-1 selected.

Mapping Active Directory Attributes to Kaleo

Now you will create Claims that will map a user’s information in your AD system to attributes that will be automatically created in the user’s Kaleo profile when they log in to Kaleo for the first time. Kaleo requires that the user’s email address be passed as the NameID. Kaleo also supports the following attributes:

  • Firstname
  • Lastname
  • Title
  • Department
  • Location
  • Mobile Phone

The Windows Event Viewer will show you any errors encountered during ADFS operations.
